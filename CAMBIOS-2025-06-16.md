# Cambios realizados — 16 Jun 2026

Resumen de todos los cambios aplicados para hacer funcionar el flujo de autenticación completo entre `eduLLM-Gateway`, `ms-auth`, `ms-profesor` (backend) y `eduLLM-Front-Profesor`.

## 1. eduLLM-Front-Profesor (Frontend del Profesor)

### `src/utils/auth.js`
- Se eliminó la dependencia a un login interno.
- Se agregó `getLoginUrl()` que lee `VITE_AUTH_URL` o construye la URL desde `VITE_GATEWAY_URL`.
- Se agregó `redirectToLogin()` que redirige al ms-auth sin limpiar cookies (no hay sesión que limpiar).
- Se agregó `logoutAndRedirect()` que primero hace `POST /api/auth/logout` al Gateway para invalidar la cookie httpOnly, y luego redirige al ms-auth.

```js
export const logoutAndRedirect = async () => {
  try {
    await fetch(`${GATEWAY_URL}/api/auth/logout`, { method: 'POST', credentials: 'include' });
  } catch {}
  localStorage.removeItem('jwtToken');
  window.location.href = getLoginUrl();
};
```

### `src/App.jsx`
- Se eliminó la ruta interna `/login` y el componente `LoginForm`.
- Se implementó el componente `AuthGate` con tres estados:
  1. **Sin sesión** → `redirectToLogin()` (sin logout, no hay cookie).
  2. **Sesión válida pero rol incorrecto** → panel "Acceso restringido" visible 3 segundos → `logoutAndRedirect()`.
  3. **Sesión válida con `ROLE_PROFESOR`** → renderiza la app normalmente.
- El rol aceptado es `ROLE_PROFESOR` o `profesor` (compatibilidad con ambas convenciones).
- El panel de rol incorrecto muestra exactamente: *"No tienes acceso a esta pagina por favor ingresa con tu usuario y contraseña."*

### `.env`
```env
VITE_GATEWAY_URL=http://localhost:8085
VITE_AUTH_URL=http://localhost:8085/login
VITE_SKIP_AUTH_VERIFY=false
```

### `CLAUDE.md`
- Documentación actualizada con arquitectura de auth, flujo de `AuthGate`, variables de entorno y archivos clave.

## 2. eduLLM-Gateway

### Bug raíz #1 — Todas las rutas eran públicas (`config/application.yml`)
**Problema:** `/` estaba en `public-paths`. El filtro evalúa `path.startsWith("/")` que es `true` para CUALQUIER ruta, haciendo que todas bypaseen la autenticación y nunca se inyectaran los headers `X-User-*`.

**Fix:** Se eliminó `/` de `public-paths`.

```yaml
# ANTES
public-paths:
  - /login
  - /
  - /css/
  ...

# DESPUÉS
public-paths:
  - /login
  - /css/
  ...
```

### Bug raíz #2 — Spring Boot normalizaba las claves del mapa a minúsculas (`config/application.yml`)
**Problema:** `@ConfigurationProperties` con `Map<String, List<String>>` normaliza las claves YAML a lowercase. La clave `ROLE_PROFESOR` quedaba almacenada como `role_profesor`, haciendo que `roleRules.get("ROLE_PROFESOR")` retornara `null` → `isAuthorized` devolvía `false` → 403 "Sin permiso".

**Fix:** Se envolvieron las claves en notación `[]` para que Spring Boot las preserve tal cual:

```yaml
# ANTES
role-rules:
  ROLE_PROFESOR:
    - "/api/profesor/**"

# DESPUÉS
role-rules:
  "[ROLE_PROFESOR]":
    - "/api/profesor/.+"
```

### Bug raíz #3 — Regex roto para rutas con sub-segmentos (`config/application.yml`)
**Problema:** El código Java hace:
```java
p.replace("**", ".*").replace("*", "[^/]*")
```
El segundo `replace` también modifica el `*` de `.*`, convirtiendo `.*` en `.[^/]*`. Así, `/api/profesor/**` terminaba como patrón `/api/profesor/.[^/]*`, que no matcheaba rutas como `/api/profesor/dashboard/graficas` (con sub-segmentos).

**Fix:** Se reemplazaron los wildcards directamente por expresiones Java regex válidas en el YAML, evitando la transformación problemática:

```yaml
# ANTES
"[ROLE_PROFESOR]":
  - "/api/profesor/**"     # → /api/profesor/.[^/]*  (roto)
  - "/api/prompting/professor/*"

# DESPUÉS
"[ROLE_PROFESOR]":
  - "/api/profesor/.+"          # cualquier sub-ruta
  - "/api/prompting/professor/[^/]+"  # un solo segmento
```

### Bug Docker Desktop / WSL2 — Bind mount roto (`Dockerfile` y `docker-compose.yml`)
**Problema:** `./config:/app/config` fallaba con `I/O error` en Docker Desktop con WSL2, dejando el contenedor sin configuración.

**Fix en `Dockerfile`:** Se bake-a la config directamente en la imagen:
```dockerfile
COPY ./target/Gateway-0.0.1-SNAPSHOT.jar app.jar
COPY ./config /app/config
```

**Fix en `docker-compose.yml`:** Se eliminaron todos los volúmenes bind-mount:
```yaml
# ELIMINADO:
volumes:
  - ./config:/app/config
  - ./logs:/app/logs
```

### Fix adicional en rutas Gateway (`config/application.yml`)
- Predicado `ms-profesor` corregido de `Path=/api/profesor/dashboard/` (trailing slash) a `Path=/api/profesor/**`.
- `role-rules` para `ROLE_PROFESOR` simplificado a `"/api/profesor/.+"`.

## 3. eduLLM-Profesor-BACK-dashboard (Backend del Profesor)

### Bug — rol incorrecto en `requireProfesor.js`
**Problema:** El middleware comparaba `req.user.rol !== 'PROFESOR'`, pero el Gateway inyecta el rol tal como viene del JWT: `ROLE_PROFESOR`.

**Fix (aplicado por el equipo):**
```js
// DESPUÉS — acepta ambas convenciones
const normalizedRol = req.user.rol?.replace(/^ROLE_/, '');
if (normalizedRol !== 'PROFESOR') {
  return res.status(403).json({ success: false, message: 'Acceso denegado: solo profesores' });
}
```

### Bug Docker Desktop / WSL2 — Bind mount roto (`Dockerfile` y `docker-compose.yml`)
**Problema:** `.:/app` montaba el directorio equivocado (ejecutado desde otra ruta), dejando `/app` vacío → `Cannot find module '/app/server.js'`.

**Fix en `Dockerfile`:**
```dockerfile
COPY package*.json ./
RUN npm install
COPY . .   # bake del código fuente en la imagen
```

**Fix en `docker-compose.yml`:** Se eliminó `.:/app`:
```yaml
volumes:
  - /app/node_modules   # solo el volumen anónimo de node_modules
```

**Se creó `.dockerignore`:**
```
node_modules
.env
logs
*.log
```

### Bug crítico — Base de datos equivocada (`docker-compose.yml`)
**Problema (identificado):** `ms-profesor` conectaba a `edu_llm` (con guion bajo), mientras que `ms-auth` conecta a `edullm` (sin guion bajo). Son dos instancias distintas de PostgreSQL con datos diferentes:
- En `edullm`: `profesor1` tiene `id_usuario=3` (el que va en el JWT).
- En `edu_llm`: `profesor1` tiene `id_usuario=22` y `tbl_m_rol` está vacía (por eso `fn_login` retornaba vacío).

**Fix pendiente** — cambiar en `docker-compose.yml`:
```yaml
# ANTES
- DATABASE_URL=postgresql://admin:admin@host.docker.internal:5432/edu_llm?schema=comun

# DESPUÉS
- DATABASE_URL=postgresql://admin:admin@host.docker.internal:5432/edullm?schema=comun
```

### Fix en BD — registro de profesor inactivo
**Problema:** En `edu_llm` (la BD usada temporalmente), el registro de `tbl_m_profesor` para `usuario_id=22` tenía `estado=false`.

**Fix aplicado temporalmente (sobre `edu_llm`):**
```sql
UPDATE comun.tbl_m_profesor SET estado = true WHERE id_profesor = 10;
```
*Este fix pierde relevancia una vez que se corrija el `DATABASE_URL` para apuntar a `edullm`.*
